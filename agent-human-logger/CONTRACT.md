# Human Logger — Contract

This file defines what this agent promises to return. The coordinator is built to expect exactly this shape.

This is A1 — the first agent in the pipeline. It receives the raw human message and is the only agent that generates an input_id. Every downstream agent must carry that input_id so every finding is traceable back to this exchange.

---

## The input

Raw human message text. No wrapping required. The coordinator sends the human's message directly.

---

## The output shape

```json
{
  "agent": "human-logger",
  "input_id": "a UUID4 generated fresh for this exchange",
  "status": "logged",
  "input": "the exact human message verbatim",
  "timestamp": "ISO 8601 UTC timestamp"
}
```

---

## Field by field

`agent` is always the string "human-logger".

`input_id` is a UUID4 generated by this agent for this specific exchange. It is the single identifier that ties all downstream verdicts together. The coordinator must extract this value and pass it to every subsequent agent.

`status` is always "logged". This agent does not fail silently — if it cannot log, it does not return.

`input` is the exact human message, character for character. No trimming, no summarising.

`timestamp` is the UTC time at the moment of logging in ISO 8601 format.

---

## Why input_id lives here

A1 is the entry point. Generating the ID here means there is exactly one ID per exchange and it exists from the first moment. All agents downstream receive it as input and return it unchanged — they do not generate IDs, they carry this one.
